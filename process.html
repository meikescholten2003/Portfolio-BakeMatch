Development Process

A look at how BakeMatch came to life, from initial concept and branding to final implementation and user testing.

Project Journey

The BakeMatch development process followed a structured UX/UI design approach combined with front-end development. Each phase built on insights from the previous one, ensuring the final product addressed real user needs while maintaining a clear, accessible design.

Concept and Branding (Step 1-2)

The project started by clearly defining the problem: many people have ingredients at home but don't know what to make with them. Traditional recipe platforms focus on ideal recipes instead of real kitchens, creating frustration.

Key insight: Users wanted to start with what they have rather than shopping for new ingredients. The focus became reversing the traditional recipe search process.

During this phase, the brand name "BakeMatch" was chosen through a systematic naming exploration. Multiple options were compared based on clarity, friendliness, and relevance to the concept. A logo was developed using AI generation guided by specific prompts describing the desired style and mood.

User personas were created representing students and young professionals with limited budgets and time. A moodboard defined the visual direction using warm beige and brown tones to create a cozy, food-related feeling.

Research and Feature Planning (Step 3-4)

A benchmark analysis was conducted on four existing platforms: SuperCook, Tasty, Lekker & Simpel, and Boodschappen.nl. Each platform was tested to identify strengths, weaknesses, and patterns in how they solve similar problems.

SuperCook confirmed that ingredient matching is a strong foundation but showed where usability could be improved. Tasty provided design inspiration for recipe presentation. Lekker & Simpel demonstrated the value of simplicity for beginner cooks. Boodschappen.nl showed how filtering can improve usability.

Based on these insights, a core feature list was defined: ingredient filter, one-missing-ingredient filter, favorites functionality, recipe database integration, AI chatbot assistance, and responsive desktop UI.

User flows were mapped out in detail, showing how users would navigate through the platform from entry point to action. This included ingredient selection, recipe browsing, favorites management, and AI assistant access.

Design and Prototyping (Step 5-10)

The design phase began with hand-drawn wireframes exploring different homepage layouts. Multiple structural variations were sketched to test navigation placement, ingredient selection positioning, and content hierarchy.

The strongest wireframe structure was then translated into a low-fidelity prototype in Figma. This black-and-white prototype focused purely on layout and user flow without visual styling. User testing at this stage revealed that the structure was clear and navigation felt intuitive.

A comprehensive style guide was created defining typography, color palette, spacing rules, icons, and UI components. The warm beige and brown color scheme supports the cozy, food-related atmosphere. Rounded shapes and friendly typography create an approachable feel.

The high-fidelity prototype brought the design to life with full colors, imagery, and branding. User testing with four participants validated that the ingredient filter was easy to understand, navigation was clear, and recipe browsing felt natural.

Development and Implementation (Step 11-19)

Development started with learning HTML and CSS fundamentals through tutorials. The core HTML structure was built first, establishing the homepage, recipe pages, and navigation system.

Instead of creating a custom database, the decision was made to use TheMealDB API, a real online recipe database. This external API provides recipe data including categories, ingredients, instructions, and images. Working with a real database made the project more realistic and helped develop skills in API integration using fetch requests.

JavaScript was used to add interactivity. The ingredient filter was implemented to dynamically show and hide recipes based on selected ingredients. The favorites functionality uses LocalStorage to save recipes directly in the browser without requiring user accounts.

An AI chatbot was integrated to help users with cooking questions and ingredient substitutions. The chatbot appears as a floating panel accessible from any page, providing support without disrupting the browsing experience.

Testing and Final Polish (Step 20)

Final user testing focused on the complete working prototype. Feedback confirmed that core features worked as expected and the visual design created the intended warm, approachable atmosphere.

UI polish included refining spacing, improving visual hierarchy, and ensuring consistency across all pages. The interface was optimized for desktop viewing while maintaining clear, accessible design patterns.

Documentation was completed, including a detailed showcase report covering every step of the process, design decisions, challenges faced, and lessons learned. An online portfolio was created to present the project professionally.

Important Decisions

Several key decisions shaped BakeMatch's direction and determined its success.

Using an External Recipe Database

Decision: Integrate with TheMealDB API instead of creating a custom database.

Reasoning: Using a real online database provides reliable, diverse recipe data without manual maintenance. It also reflects how real recipe platforms work in practice and develops valuable API integration skills.

Outcome: The platform has access to a large, professional recipe database with high-quality images and detailed instructions, making it immediately useful and realistic.

Warm, Cozy Visual Design

Decision: Use beige and brown tones with rounded shapes instead of bright, energetic colors.

Reasoning: The warm color palette creates a cozy, food-related feeling that makes cooking feel approachable and non-intimidating for beginners. This supports the platform's goal of accessibility.

Outcome: Users respond positively to the friendly atmosphere. The design feels like a helpful companion in the kitchen rather than an intimidating technical tool.

No User Accounts Required

Decision: Use LocalStorage for favorites instead of requiring user registration.

Reasoning: Creating accounts creates friction for new users. LocalStorage allows the favorites feature to work immediately without barriers, fitting the project scope while still providing value.

Outcome: Users can start using BakeMatch immediately and save recipes without any setup process. This reduces the barrier to entry significantly.

Desktop-First Approach

Decision: Design and optimize primarily for desktop viewing.

Reasoning: The project scope focused on desktop implementation. Users often cook with a laptop or tablet on the counter, making desktop optimization appropriate for the use case.

Outcome: The interface has enough space to show the ingredient filter, recipe grid, and navigation clearly without feeling cramped or requiring excessive scrolling.

Challenges and Solutions

Challenge: Choosing the Right Visual Direction

Problem: Multiple UI styles could work for a recipe platform. Finding one that truly fits BakeMatch's goal of accessibility was challenging.

Solution: Compared different styles by asking which feels less intimidating and better supports beginner cooks. A warm, cozy style was chosen over clean minimal or playful illustrated designs because it creates the right emotional tone.

Challenge: Creating Realistic Personas

Problem: Early personas felt too generic and didn't provide enough specific guidance for design decisions.

Solution: Researched well-built persona examples to understand what information is essential. Focused on motivations, frustrations, and real-life situations instead of just demographics. This made personas more useful for validating feature choices.

Challenge: Balancing Visual Detail in Wireframes

Problem: Deciding how detailed wireframes should be without becoming visual design.

Solution: Kept wireframes strictly black and white with no imagery. This forced focus on structure, hierarchy, and usability. Visual styling was saved for the high-fidelity prototype phase, creating a clear separation of concerns.
